/**
 * Aggregates (Агрегаты)
 * 
 * Эта директория содержит агрегаты - кластеры связанных объектов, которые
 * рассматриваются как единое целое с точки зрения изменений данных.
 * Агрегаты инкапсулируют сложную бизнес-логику и обеспечивают целостность данных.
 * 
 * ## Основные обязанности:
 * - Инкапсуляция сложной бизнес-логики и правил целостности
 * - Управление транзакционной согласованностью группы объектов
 * - Предоставление единой точки доступа для изменений данных
 * - Обеспечение инвариантов (business invariants) внутри агрегата
 * - Управление жизненным циклом вложенных сущностей и value objects
 * 
 * ## Ключевые компоненты:
 * 
 * ### Aggregate Roots (Корни агрегатов)
 * Основные сущности, которые являются точкой входа в агрегат:
 * - Содержат глобальный идентификатор
 * - Инкапсулируют бизнес-правила и инварианты
 * - Управляют вложенными сущностями и value objects
 * - Публикуют доменные события при изменениях
 * 
 * ### Вложенные сущности
 * Сущности, которые существуют только в контексте агрегата:
 * - Не имеют глобальной идентичности
 * - Идентифицируются относительно корня агрегата
 * - Подчиняются бизнес-правилам агрегата
 * 
 * ## Особенности агрегатов:
 * - Имеют четкие границы (aggregate boundaries)
 * - Корень агрегата - единственная точка доступа извне
 * - Ссылки между агрегатами только по идентификаторам
 * - Изменения внутри агрегата атомарны
 * - Инварианты всегда соблюдаются внутри агрегата
 * 
 * ## Примеры агрегатов:
 * 
 * ### Order Aggregate
 * - Корень: Order
 * - Вложенные: OrderItem, Payment, ShippingAddress
 * - Инварианты: сумма завода должна равняться сумме позиций
 * 
 * ### User Profile Aggregate
 * - Корень: UserProfile
 * - Вложенные: ContactInfo, Preferences, SecuritySettings
 * - Инварианты: email должен быть уникальным и валидным
 * 
 * ### Shopping Cart Aggregate
 * - Корень: ShoppingCart
 * - Вложенные: CartItem, Discount, ShippingOption
 * - Инварианты: корзина не может содержать отрицательное количество
 * 
 * ## Best Practices:
 * - Проектируйте небольшие агрегаты с четкими границами
 * - Инкапсулируйте бизнес-правила внутри агрегатов
 * - Используйте идентификаторы для ссылок между агрегатами
 * - Публикуйте доменные события для меж-агрегатного взаимодействия
 * - Тестируйте инварианты и бизнес-правила агрегатов
 * - Следите за целостностью данных при изменениях
 * - Документируйте границы и ответственность каждого агрегата
 * - Избегайте больших агрегатов с множеством вложенных сущностей
 * 
 * ## Принципы проектирования:
 * 1. **Единство изменений** - агрегат изменяется как единое целое
 * 2. **Границы транзакций** - агрегат определяет границы транзакций
 * 3. **Ссылочная целостность** - ссылки между агрегатами по ID
 * 4. **Событийное взаимодействие** - агрегаты общаются через события
 * 5. **Инвариантная защита** - бизнес-правила всегда соблюдаются
 */
